# Безопасная запись BSL через `write_module_source`

> Самая опасная зона. Неправильный выбор режима может **перезаписать весь модуль** или потерять часть кода. Читай этот файл **до** первой записи.

## Доступные режимы

`write_module_source` поддерживает три режима (`mode`):

| Режим | Что делает | Когда применять |
|---|---|---|
| `searchReplace` (default) | Находит `oldSource` в файле и заменяет на `source`. Безопасный — затрагивает только то, что нашёл | **Дефолт для любой точечной правки** |
| `append` | Дописывает `source` в конец существующего модуля | Когда нужно добавить новый метод/секцию в конец |
| `replace` | **Перезаписывает весь файл модуля** содержимым `source` целиком | Создание нового модуля или полная переработка |

Поведенческие гарантии сервера:
- При `searchReplace` `oldSource` должен **встретиться в файле ровно один раз**. Если 0 совпадений — операция отклонена (`oldSource not found`), если >1 — также отклонена с подсказкой «расширь фрагмент». Эта проверка спасает от случайной замены не в том месте.
- При `append`/`searchReplace` файл должен уже существовать. Создать новый `.bsl` можно **только** через `replace`.
- Перед записью переводы строк `\r\n` нормализуются в `\n`. UTF-8 BOM сохраняется (а для нового файла — добавляется).
- Лимит на размер `source` — 500 000 символов.
- В `modulePath` запрещён `..`; принимаются только пути на `.bsl`-файлы.

## Правило №1: НИКОГДА не используй `replace` для одного метода

`replace` заменяет **весь файл модуля**. Если передать туда только один метод — остальной код (сотни/тысячи строк) будет удалён без возможности отката.

Для одного метода — `searchReplace`: в `oldSource` помести существующий текст метода (должен точно совпадать с одним местом в файле), в `source` — новую версию. Если метод длинный — разбей правку на логические блоки и сделай несколько вызовов.

## Правило №2: Перед боевой записью — посмотри состояние

1. Прочитай метод через `read_method_source` (или диапазон через `read_module_source`) — увидишь актуальное состояние.
2. Через `get_module_structure` оцени размер модуля (число процедур/функций, диапазоны строк).
3. Только после этого формируй вызов `write_module_source`.

`oldSource` в `searchReplace` дополнительно служит **доказательством**, что ты прочитал текущее содержимое файла. Если файл изменился — операция упадёт, и это правильно.

## Правило №3: Когда `replace` допустим

- Модуль создаётся **впервые** (файла ещё нет — только `replace` может его создать).
- Модуль пустой или меньше ~50 строк, и ты передаёшь его полное содержимое.
- Полная переработка модуля **после явного согласования с пользователем**.

## Выбор модуля: `modulePath` vs `objectName + moduleType`

Можно указать либо `modulePath` (путь от `src/`, например `Documents/MyDoc/ObjectModule.bsl`), либо пару `objectName + moduleType`. Если задан `modulePath` — он имеет приоритет.

`moduleType` (используется с `objectName`):
- `ObjectModule` (по умолчанию для большинства типов: справочники, документы и т. п.)
- `ManagerModule`
- `RecordSetModule` (для регистров)
- `FormModule` — **обязателен** `formName` (например, `ItemForm`); для `CommonForm` `formName` не нужен.
- `CommandModule` — **обязателен** `commandName`; для `CommonCommand` `commandName` не нужен.

Особые случаи дефолта `moduleType` (если не указан явно):
- `CommonModule`, `CommonForm`, `WebService`, `HTTPService` → внутренне ведёт к файлу `Module.bsl`.
- `CommonCommand` → `CommandModule.bsl`.
- Всё остальное → `ObjectModule.bsl`.

Если правишь общий модуль — проще указывать `modulePath: CommonModules/<Name>/Module.bsl`, чтобы не зависеть от логики дефолтов.

## Что проверяет встроенный синтаксический контроль

По умолчанию `skipSyntaxCheck=false` — сервер проверяет **полный итоговый файл** (после применения правки) на **баланс открывающих/закрывающих ключевых слов BSL**:

- `Процедура` / `КонецПроцедуры` (`Procedure` / `EndProcedure`)
- `Функция` / `КонецФункции` (`Function` / `EndFunction`)
- `Если` / `КонецЕсли` (`If` / `EndIf`)
- `Пока` / `КонецЦикла` (`While` / `EndDo`)
- `Для` / `КонецЦикла` (`For` / `EndDo`)
- `Попытка` / `КонецПопытки` (`Try` / `EndTry`)

Регистр игнорируется, русский и английский — равноправно. Чекер пропускает пустые строки, цельные `//`-комментарии, строки-продолжения многострочных строк (начинаются с `|`). Внутрь выражений и в правильность вызовов проверка **не лезет** — это не полный парсер, а быстрая проверка баланса блоков.

Чего проверка **не поймает**: опечатку в имени функции, неверное число аргументов, обращение к несуществующему методу, неверный тип, неэкранированную кавычку внутри строкового литерала, забытую `;` в выражении. Эти ошибки обнаружит `get_project_errors` после записи.

`skipSyntaxCheck=true` ставь только когда сам быстрый чекер выдаёт ложный позитив (редкий случай) и ты вручную убедился, что баланс блоков на самом деле в порядке.

## Алгоритм безопасной правки метода

1. `read_method_source` — прочитай метод. Запомни `startLine`/`endLine`.
2. Подготовь исправленный код метода целиком (от `Процедура`/`Функция` до `КонецПроцедуры`/`КонецФункции`, включая директивы `&НаКлиенте`, `&НаСервере`, `&НаСервереБезКонтекста`).
3. Вызови `write_module_source` с:
   - `mode: "searchReplace"`,
   - `oldSource` — текст старого метода ровно как в файле (точное совпадение, единственное в файле),
   - `source` — твоя новая версия.
4. Если сервер вернул `oldSource not found` или `multiple times` — **не повторяй вслепую**: перечитай файл и либо обнови `oldSource`, либо расширь его так, чтобы он стал уникальным.
5. После успешной записи проверка баланса блоков уже выполнена сервером. Дополнительно вызови `get_project_errors` с фильтром `objects` на изменённый модуль, чтобы убедиться, что не появились семантические ошибки.

## Алгоритм добавления нового метода

1. Через `get_module_structure` посмотри, в какой регион (`#Область ПрограммныйИнтерфейс` / `СлужебныйПрограммныйИнтерфейс` / `СлужебныеПроцедурыИФункции`) попадёт метод.
2. Вариант А (если регион в конце модуля или метод служебный): `mode: "append"` — добавит в конец файла.
3. Вариант Б (нужно вставить в середину или конкретный регион): `mode: "searchReplace"`, ищи маркер начала региона / закрывающую строку — заменяй на «маркер + новый метод».

## Что делать после записи

- Если в ответе сервера присутствует список ошибок баланса блоков — исправь их **до** перехода к следующей правке.
- Запусти `get_project_errors` с фильтром `objects: ["<FQN объекта>"]` для подтверждения, что новых проблем нет.
- Если правок несколько подряд — после серии вызови `get_problem_summary`.

## Откат при неудачной правке

В EDT-MCP нет встроенного undo для `write_module_source`. Откат держится на том, что **до** записи ты помнишь, какой код там был.

1. **Перед каждым `searchReplace` сохрани в контексте сессии `oldSource`, который ты отправляешь**, и `source` (новую версию). Не сбрасывай их, пока не убедился, что запись корректна.
2. Если по ответу видно, что заменён не тот фрагмент или сломалась структура модуля — **немедленно** выполни обратный `searchReplace`: в `oldSource` поставь то, что было `source`, в `source` — то, что было `oldSource`. Чем быстрее, тем меньше шансов, что параллельный редактор EDT увидит сломанный файл.
3. Если откатить через `searchReplace` уже невозможно (новый `source` не уникален в файле, или модуль перешёл через `replace`) — **остановись и спроси пользователя**. Не пытайся восстанавливать модуль «по памяти»; в EDT может быть открытая копия, и через UI восстановление надёжнее.
4. Для режима `replace` правила те же: сохраняй полный старый текст до отправки нового, иначе откат невозможен.

## Чего категорически не делать

- Не редактировать `.bsl` напрямую через `Edit`/`Write`/`Read+Write`. Только через `write_module_source` — иначе EDT не увидит изменений до перечитывания workspace, а ты потеряешь автоматическую проверку и подсчёт строк.
- Не использовать `replace` без полного содержимого нового модуля.
- Не передавать в `write_module_source` усечённое содержимое, надеясь «сервер сам разберётся» — он перезапишет ровно то, что ты прислал.
- Не выставлять `skipSyntaxCheck=true`, чтобы «обойти» ошибки баланса блоков — это почти всегда симптом реальной ошибки в коде.
- Не коммитить изменения, если `get_project_errors` показывает новые проблемы в изменённом модуле.
